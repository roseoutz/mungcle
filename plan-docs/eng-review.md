## 핵심 기술 결정

### 200m Grid Snap (ADR-005, 불변)
- GPS 좌표를 200m 그리드 중심점으로 스냅
- 정확한 좌표는 저장/전송하지 않음 (프라이버시)
- 구현: `floor(lat / 0.002) * 0.002` (간단한 수학, PostGIS 불필요)
- GPS 거부 시: 드롭다운으로 수동 동네/공원 선택

### 인사하기 플로우 (MSA 버전)
```
1. [api-gateway] POST /api/greetings → gRPC → social-service
2. [social] identity.checkBlocked(sender, receiver) → gRPC
3. [social] Greeting(PENDING) 저장
4. [social] Kafka publish: greeting.created
5. [notification] Kafka consume → FCM 푸시 to receiver
6. [api-gateway] POST /api/greetings/:id/respond → gRPC → social-service
7. [social] Greeting → ACCEPTED, 30분 만료 설정
8. [social] Kafka publish: greeting.accepted
9. [notification] Kafka consume → FCM 푸시 to sender
10. [social] @Interval: 5분 무응답 → EXPIRED, Kafka publish
```

### Coroutine vs Virtual Thread (ADR-016)
- **Virtual Thread:** api-gateway REST, JPA/JDBC, Kafka Consumer
- **Coroutine:** gRPC 서버 핸들러, Kafka Producer, 서비스 내부 비동기 조합
- 원칙: Blocking I/O → VT, Non-blocking → Coroutine

### TSID (ADR-017)
- 64-bit Long. `@Tsid` 어노테이션 (hypersistence-utils).
- Node ID는 서비스 이름 기반 자동 할당.
- MSA에서 서비스별 독립 ID 생성, 충돌 구조적 불가능.

## 데이터 모델

서비스별 스키마 분리. Cross-schema JOIN 금지.

### identity schema
```
User: id(TSID), kakaoId?, email?, passwordHash?, nickname, neighborhood?,
      pushToken?, flaggedForReview, deletedAt?, createdAt
Block: id, blockerId(→User), blockedId(→User), createdAt
       UNIQUE(blockerId, blockedId)
Report: id, reporterId(→User), reportedId(→User), reason, createdAt
```

### pet_profile schema
```
Dog: id(TSID), ownerId(외부 User TSID 참조), name, breed, size(SMALL|MEDIUM|LARGE),
     temperaments(String[]), sociability(1-5), photoPath?, vaccinationPhotoPath?,
     deletedAt?, createdAt
```

### walks schema
```
Walk: id(TSID), dogId(외부), userId(외부), type(OPEN|SOLO), gridCell,
     status(ACTIVE|ENDED), startedAt, endsAt, endedAt?, createdAt
     INDEX(gridCell, status, startedAt DESC)
     INDEX(dogId, status)
WalkPattern: id, gridCell, hourOfDay, dogId(외부), walkCount, lastWalkedAt
```

### social schema
```
Greeting: id(TSID), senderUserId(외부), receiverUserId(외부), senderDogId(외부),
          receiverDogId(외부), receiverWalkId(외부), status(PENDING|ACCEPTED|EXPIRED),
          createdAt, respondedAt?, expiresAt
          UNIQUE(senderUserId, receiverUserId, receiverWalkId)
Message: id(TSID), greetingId(→Greeting), senderUserId(외부), body, createdAt
```

### notification schema
```
Notification: id(TSID), userId(외부), type(GREETING_RECEIVED|GREETING_ACCEPTED|
              MESSAGE_RECEIVED|WALK_EXPIRED), payload(JSONB), readAt?, createdAt
              INDEX(userId, readAt, createdAt DESC)
```

> `(외부)` = 다른 서비스 소유 엔티티의 TSID. JOIN 불가, gRPC로 조회.

## Error Handling

| Codepath | Failure Mode | Handling | User Sees |
|----------|-------------|----------|-----------|
| 카카오 OAuth | API 장애 | 이메일 fallback | "카카오 로그인 실패, 이메일로 진행" |
| 사진 업로드 | Supabase 장애 | retry + 메시지 | "사진 업로드 실패, 다시 시도" |
| GPS 위치 | 권한 거부 | 수동 동네 선택 | 동네 선택 드롭다운 |
| gRPC 서비스간 | 대상 서비스 다운 | Circuit Breaker + fallback | "잠시 후 다시 시도" |
| Kafka publish | 브로커 장애 | retry + DLQ | 알림 지연 (데이터 유실 없음) |
| nearby 쿼리 | DB 타임아웃 | 500 + retry | "잠시 후 다시 시도" |
| 인사 푸시 | FCM 전송 실패 | 인앱 알림 fallback | 앱 열면 확인 |
| grid snap | 경계값 | 순수 계산, N/A | 잘못된 cell 배정 (테스트로 방지) |
| CRON 만료 | @Interval 미실행 | health check | 산책 상태 안 꺼짐 |

## 성능 고려사항

- **N+1 방지:** JPA `@EntityGraph` 또는 `JOIN FETCH` 사용
- **CRON:** 서비스별 `@Scheduled(fixedRate = 60_000)`. Virtual Thread 위에서 실행.
- **시간대 패턴:** 초기 50명 규모에서 직접 쿼리 OK. 데이터 쌓이면 사전 집계 테이블.
- **gRPC 호출 최적화:** nearby 쿼리 시 pet-profile bulk 조회 (`getDogsByIds`), identity 차단 목록 bulk 조회 (`getBlockedUserIds`).
- **동시 접속:** MVP 규모 (50-200명) 성능 이슈 없음.

## 구현 병렬화 전략

```
Lane A: 프로젝트 scaffolding + proto 정의 + common 패키지         ← 반드시 먼저
Lane B: identity-service (auth + users + blocks + reports)        ← A 완료 후
Lane C: pet-profile-service (dogs)                                ← B 완료 후 (User 필요)
Lane D: walks-service (walks + patterns)                          ← B 완료 후 병렬
Lane E: social-service (greetings + messages)                     ← B + D 완료 후
Lane F: notification-service (kafka consumer + FCM)               ← B 완료 후 병렬
Lane G: api-gateway (REST BFF + gRPC client)                      ← B~F와 병렬, proto 확정 후
Lane H: Expo app shell + navigation                               ← G의 REST API 계약 후 병렬

실행 순서:
  1. A (foundation) — scaffolding, proto, docker-compose
  2. B (identity) — 인증 기반
  3. C + D + F + G 병렬
  4. E (social) — D(walks) 완료 후
  5. H (frontend) — G(gateway) REST 안정화 후
  6. 통합 테스트
```

**충돌 포인트:** 각 서비스가 별도 스키마를 사용하므로 Flyway 마이그레이션 충돌 없음 (모놀리스 대비 장점).

## 배포 전략

- **개발:** Docker Compose (로컬). 모든 서비스 + 인프라 한 번에 기동.
- **MVP 테스트:** Docker Compose + ngrok 터널링 → TestFlight 앱에서 접근.
- **정식 배포:** Go/No-go 통과 후 Fly.io 또는 Railway. 서비스별 컨테이너 이미지.
- **DB:** Supabase free tier (500MB). 서비스별 스키마.
- **앱:** EAS Build → TestFlight (iOS) / 내부 테스트 (Android).
- **앱스토어 정식 배포:** Phase 1 go/no-go 통과 후.
