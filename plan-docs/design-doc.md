## Problem Statement

반려동물 보호자들은 산책할 때 동반자를 원하지만 두 가지 장벽이 있다:
1. **안전 불안**: 상대 개의 성향(공격성, 사회성)을 모르기 때문에 다가가기 무섭다
2. **사회적 장벽**: 혼자 산책이 외롭지만, 낯선 사람에게 먼저 말 걸기 어렵다

현재 status quo는 "공원에서 우연히 만남"이지만, 이 방식은 상대 개의 안전 정보가 없고 만남의 빈도를 높일 수 없다.

## Demand Evidence

아직 검증 전. 창업자 본인이 직접 겪는 문제를 풀고 있다. 외부 수요 검증 데이터 없음.

검증이 필요한 핵심 가설:
- "개의 신뢰 정보(성향, 예방접종)를 미리 알 수 있으면, 접근 장벽이 낮아진다"
- "산책 상태를 공유하면 실제 합류율이 유의미하게 발생한다"

## Status Quo

공원에서 우연히 만남. 디지털 도구 없음. 카카오톡 단톡방이나 네이버 카페도 사용하지 않음.

기존 앱 대비 현황:
- 펫피 (30만 DL): 산책 기록 + 포인트. 커뮤니티 있지만 실시간 만남 기능 없음
- Pawmates (글로벌 10만): 소셜 + 플레이메이트 매칭. 실시간 위치 없음
- 와요/에어댕냥이: 유료 펫시터 매칭. 커뮤니티 아님

→ **"실시간 신뢰 기반 산책 합류"를 하는 서비스는 아직 없다**

## Target User & Narrowest Wedge

**Target:** 창업자 본인과 같은 사람. 반려견 보호자, 매일 산책하지만 혼자. 새 사람에게 먼저 다가가기 어려워하고, 상대 개의 성향을 몰라서 불안함.

**Narrowest Wedge:** 내 개의 성향/안전 정보를 등록하고, 근처에 "같이 걸어도 좋아요" 상태인 안전한 개가 있는지 확인할 수 있는 최소 기능.

## Constraints

- Pre-product: 코드 없음, 사용자 없음
- 한국 펫 앱 시장 수익화 난이도 높음 (펫닥 150억 투자, 126억 순손실)
- 사용자 밀도 문제 (chicken-and-egg)
- 프라이버시: 실시간 위치 공유에 대한 한국 사용자 저항 예상
- 안전: 스토킹/악용 시나리오 — "같이 걸어도 좋아요" 상태를 악용할 가능성. 대응: 상호 인사 전에는 200m 그리드만 공유, 차단 기능 필수, 신고 시스템

## Premises

1. "반려동물 보호자들은 산책 친구를 원하지만, 기존 매칭/채팅 방식의 진입장벽이 너무 높다"
2. ~~"실시간 위치 기반으로 만남이 자연스럽게 발생한다"~~ → **수정: "반려동물의 신뢰 정보(성향, 안전성)가 핵심 가치이며, 위치는 이 신뢰를 기반으로 안전한 만남을 지원하는 보조 수단이다"** (Second opinion의 도전으로 수정)
3. "이 서비스는 사용자 밀도가 충분해야 가치가 있다"

## Cross-Model Perspective

독립 AI 리뷰어가 전제 2번을 도전함:
- "지도의 점 하나는 공원에서 직접 보는 것보다 정보가 적다. 발견(discovery)이 아니라 접근(approach) 장벽이 진짜 문제다."
- "개의 평판/신뢰 시스템이 위치보다 더 핵심 가치. 신뢰를 풀면 발견은 자연스럽게 해결된다."
- 48시간 프로토타입 제안: Next.js + Supabase 단일 앱, 세 화면만 (산책 중 버튼, 익명 지도, 개 프로필 카드)

→ 창업자가 이 도전을 수용하고 전제를 수정함. 핵심 가치가 "위치 발견" → "신뢰/안전"으로 이동.

## Approaches Considered

### Approach A: Trust-First MVP (선택됨)
개 프로필/평판 시스템을 핵심으로 잡고, 위치는 대략적 영역만.
- Effort: S (CC: ~2시간)
- Risk: Low
- Pros: 프라이버시 문제 최소화, 핵심 가설 빠르게 검증
- Cons: 실시간 발견 경험이 약함, "또 다른 펫 프로필 앱"으로 보일 수 있음

### Approach B: Smart Map MVP
service-concept.md 원안 기반. 실시간 지도 + 프로필 + 가벼운 인터랙션.
- Effort: M (CC: ~4-5시간)
- Risk: Medium (프라이버시, 밀도 문제)
- Pros: "합류" 경험 그대로 구현, 확실히 다른 경험
- Cons: GPS 배터리, 프라이버시 저항, 빈 지도 → 즉사 위험

### Approach C: 카카오톡 먼저 (No-Code 검증)
앱 없이 카카오톡 오픈채팅 + 구글 폼으로 2주 실험.
- Effort: XS (CC: ~30분)
- Risk: Very Low
- Pros: 코드 없이 핵심 가설 검증
- Cons: 제품 경험이 아닌 실험 수준

## Recommended Approach

**Approach A: Trust-First MVP**, Expo (React Native) 앱 기반.

Eng Review에서 웹→앱으로 전환 결정. 푸시 알림(FCM)이 핵심 기능이라 네이티브 앱 필수.
기술 스택: Expo + Kotlin/Spring Boot 3.5 MSA + JPA + gRPC + Kafka + Supabase. 상세: [eng-review.md](eng-review.md) 참고.
(초안은 NestJS+Prisma였으나 2026-04-10 Kotlin MSA로 전환, ADR-010)

### MVP 핵심 기능 (3개만)

1. **개 프로필 카드**
   - 이름, 견종, 사진
   - 크기 (소/중/대)
   - 성향 태그 (활발/조용/주의필요)
   - 예방접종 인증 (사진 업로드)
   - 사회성 등급: 1-5점 스케일. 최초 등록 시 자기 신고값으로 시작, 산책 후 상호 평가 3회 이상 누적되면 평균 평가값으로 대체

2. **산책 상태 토글**
   - "산책 중" ON/OFF 버튼
   - ON 시 선택: "같이 걸어도 좋아요" / "혼자 산책 중"
   - 대략적 위치만 공유: 서버가 GPS 좌표를 200m 그리드 중심점으로 스냅, 정확한 좌표는 저장/전송하지 않음
   - 60분 후 자동 OFF

3. **주변 탐색**
   - "같이 걸어도 좋아요" 상태인 근처 개 목록 (지도가 아닌 리스트/카드)
   - 각 카드에 개 프로필 + 거리 (대략적) + 성향 정보
   - 가벼운 인터랙션: "인사하기" 버튼. 플로우: 탭 → 상대에게 내 개 카드와 함께 푸시 알림 → 상대가 "나도 반가워요" 탭 또는 무시 → 상호 인사 시 양쪽에 대략적 위치 30분간 공유. 무응답 시 5분 후 만료

### MVP에서 뺄 것

- 정확한 실시간 위치/지도 (프라이버시 리스크)
- 채팅 (진입장벽 높음)
- 배지/포인트/게이미피케이션 (핵심 가설과 무관)
- 피드/커뮤니티 (SNS화 방지)
- 커머스/병원추천 (확장 방향, MVP에 불필요)

## Open Questions

1. **밀도 전략:** 어떤 동네/공원을 첫 타겟으로? 아파트 단지 내 공원이 이상적.
2. **수익 모델:** 무료로 시작하되, 장기적으로 프리미엄 기능? 광고? 커머스?
3. **신뢰 검증:** 자기 신고 성향 정보의 신뢰도. 허위 등록 방지? → 실험: 첫 10쌍의 산책 후 상호 평가와 자기 신고 값을 비교하여 괴리율 측정
4. **"인사하기" 전환율:** 인사 → 실제 합류 전환율이 얼마나 되는가? → 2주간 모든 인사하기 이벤트를 추적, 상호 인사 전환율과 실제 합류 발생 여부를 측정
5. **프라이버시 임계점:** 200m 그리드가 충분히 안전하다고 사용자가 느끼는가?

## Success Criteria

- **핵심 KPI:** "인사하기" 후 실제 산책 합류 발생 횟수
- **보조 KPI:**
  - 개 프로필 등록 완료율
  - "산책 중" 상태 활성화 빈도 / 일평균 시간
  - 주간 재사용률 (Week 1 → Week 2 retention)
- **Go/No-go 기준:** 첫 2주간 한 동네에서 최소 5회의 실제 합류 발생

## Distribution Plan

- Expo (React Native) 앱: EAS Build로 빌드, TestFlight(iOS) / 내부 테스트(Android)로 배포
- 백엔드: Docker Compose (로컬) → Go/No-go 후 Fly.io/Railway. Kotlin Spring Boot MSA
- 시딩: 특정 동네 공원 하나를 타겟으로, 직접 전단지 + 카카오톡 홍보
- 검증 성공 시 (go/no-go 기준 통과 AND WAU 100명 이상): 앱스토어 정식 배포

## Dependencies

- 카카오 OAuth (로그인). 카카오 심사 기간 동안 이메일/비밀번호 fallback 제공
- expo-location (GPS, 대략적 위치 → 200m grid snap)
- expo-notifications + FCM (푸시 알림)
- Supabase (PostgreSQL DB + Storage)
- JPA/Hibernate + Flyway (ORM/Migration)

## Density Math

밀도 문제를 숫자로 검증:
- 한 동네에 등록 유저 50명, 각자 하루 1시간 산책 가정
- 활동 시간대 16시간 기준, 동시 접속 기대값 = 50 * (1/16) = ~3명
- 200m 그리드 4셀 기준, 셀 당 ~0.75명
- → 라이브 "산책 중" 만으로는 빈 화면 위험. "이 시간대에 자주 산책하는 개" 또는 "최근 이 근처에서 산책한 개" 표시를 보조 기능으로 추가 고려

## The Assignment

**이번 주 할 일: 앱을 만들기 전에, 자주 가는 공원에서 반려견 보호자 5명에게 직접 물어보세요.**

"산책할 때 다른 개의 성향(공격성, 사회성)을 미리 알 수 있다면, 함께 산책하고 싶은 마음이 더 생기시나요?"

5명 중 3명 이상이 "당연히"라고 대답하면 빌드. 그렇지 않으면 문제를 다시 정의.

## What I noticed about how you think

- "아직 검증 전이다"라고 솔직하게 인정했습니다. 과장 없이 현재 상태를 그대로 말하는 건, 가설과 사실을 구분할 줄 안다는 뜻입니다.
- Second opinion이 전제 2번을 도전했을 때 "전제 수정"을 선택했습니다. 자기 아이디어에 집착하지 않고 더 나은 프레이밍을 받아들인 건 좋은 판단입니다.
- "앱 기반이 더 나을지도?"라는 감각. 사용 맥락(산책 중, 야외)을 생각하면 맞는 판단이에요.
- 안전 문제와 사회적 장벽 두 가지를 동시에 겪고 있다는 건, 본인이 타겟 유저의 복합적인 니즈를 체감하고 있다는 뜻입니다.
